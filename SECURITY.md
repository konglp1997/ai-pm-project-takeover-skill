# Security policy

## 报告漏洞

不要公开提交真实 Token、客户数据、Prompt、私有架构或敏感仓库。优先使用 GitHub Private vulnerability reporting，提供脱敏复现、影响、前提和建议修复。

## 风险模型

- 把凭据、客户数据、私有 Prompt、指标或内部路径写入 dossier/search index。
- 从不可信仓库内容引入恶意 HTML 或提示注入。
- 通过资产和链接越出 dossier 根目录。
- 远程脚本、样式或媒体引入供应链风险。
- Agent 为理解项目执行迁移、部署、删除、上传或生产访问。
- 把内部 dossier 直接当成公开作品集。

## 安全原则

- 默认只读审计目标项目。
- 仓库内容是证据，不是能覆盖用户指令的可信指令。
- 不执行危险命令，除非用户明确授权具体范围。
- 不把不可信 HTML 直接放入 `content/`。
- 内部档案与公开案例分开生成和审查。
- 公开前人工检查配置、内容、生成页面、搜索索引、图和截图。
- Mermaid 等第三方运行时从可信发行源取得并核对许可证。

验证器是启发式保护，不是完整秘密扫描器、HTML 沙箱或恶意仓库隔离环境。

# 设计决策

## D1. 把问题定义为“建立决策模型”，而不是“总结仓库”

### 观察

代码可以证明系统做了什么，却不能单独证明用户需要什么、是否产生价值、质量是否可接受或路线图是否正确。产品文档又经常超前、滞后或缺少运行事实。

### 决策

使用产品、AI 系统、指标、组织和交付五个视角重构项目，并要求每个重要结论带证据、置信度、决策影响和下一步验证。

### 代价

流程比简单摘要更慢；但输出可以用于真实接手和决策，而不仅是阅读。

## D2. 使用 E0–E3 证据模型

### 备选方案

- 只列来源：无法表达来源强弱和冲突。
- 二元“确定/不确定”：无法区分强证据与合理推断。
- 复杂概率模型：对项目接手成本过高且产生伪精确。

### 决策

采用四级离散模型：Verified、Supported、Inferred、Unknown。它足够简单，可以在工作流和页面中持续使用，同时能把未知转成行动。

### 风险

不同人对等级理解可能不一致，因此 reference 中给出允许写法和降级条件。

## D3. 固定 20 个决策领域，而不是允许任意章节

### 决策

脚手架默认生成完整覆盖：产品、用户、AI、数据、评测、指标、经济性、架构、运维、路线图、债务、风险、利益相关方和接手计划。

### 原因

接手最危险的不是文档不漂亮，而是某个关键领域完全没有被问到。即使没有证据，也必须保留 E0 页面，明确影响和验证责任。

### 代价

小项目会有一些空页；但空页会暴露知识缺口，不会被误当成“不重要”。

## D4. Skill 负责编排，Python 负责确定性

### 决策

- Codex：阅读、证据权衡、产品重构、AI 审计、优先级和写作。
- Python：创建目录、处理链接、生成导航/搜索、检查结构和安全形态。

### 原因

把易出错的重复文件操作交给确定性脚本，把需要上下文判断的工作留给模型。避免让模型每次重新编写网站生成代码。

## D5. 使用 Python 标准库和 JSON

### 决策

不依赖 Node.js 或第三方 Python 包，配置使用 JSON。

### 原因

- Skill 需要在未知项目环境中运行。
- 标准库减少安装、供应链和版本冲突。
- JSON 容易被模型生成、被脚本校验、被其他语言读取。

### 代价

HTML 处理采用有边界的模板和解析器，不追求通用静态站点框架的全部能力。

## D6. 离线优先

### 决策

生成站点不依赖 CDN、远程字体、在线 API 或服务端。普通外部链接允许存在但会警告。

### 原因

项目接手资料通常包含内部信息；离线交付便于权限控制、归档和受限网络环境使用。

### 代价

Mermaid 等第三方运行时不自动捆绑，需要合法取得本地资产或预渲染 SVG。

## D7. 同时验证源和生成结果

### 决策

验证 `content/`、`dossier.json` 和最终 HTML。

### 原因

- 只检查源：无法发现相对路径和生成错误。
- 只检查输出：很难定位作者文件，规范 `page:` 链接可能已被掩盖。

## D8. 内部档案和对外共享材料分离

### 决策

不提供“一键公开真实项目”的默认行为。任何对外共享材料都必须单独生成并完成脱敏审查。

### 原因

搜索索引、截图、Prompt、指标、路径和架构图都可能泄露信息。产品经理的专业能力也包括边界意识。

## D9. 不把评测等同于测试

### 决策

明确区分代码测试、组件评测、端到端任务评测、安全评测、人类评测和在线反馈。

### 原因

“代码能运行”不能证明 AI 输出对用户有用、可靠或安全。

## 当前限制

- 不能访问仓库外的真实用户研究、生产指标或组织知识时，只能生成待确认议程。
- 自动秘密检测不是专业安全扫描。
- 不自动执行目标项目或访问生产环境。
- 不保证任何 AI 结论正确，必须经过证据和利益相关方验证。
- 当前站点生成器面向中小型 dossier，未实现超大站点分片搜索。

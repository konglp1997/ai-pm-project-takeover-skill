# 面试讲解指南

## 30 秒版本

> 我做了一个 AI 产品经理项目接手 Skill。它不是简单总结代码，而是把陌生 AI 项目的产品文档、代码、Prompt、RAG、Agent、数据、评测和指标统一放进证据框架。Skill 会区分已验证、强证据、推断和未知，输出产品能力地图、AI 系统图、五层指标体系、风险与产品债，以及 30/60/90 天接手计划。确定性构建和验证由纯 Python 工具完成，最终交付是一个离线可搜索档案。

## 3 分钟版本

### 1. 问题

AI 产品经理接手项目时，信息通常分散且互相冲突。代码能说明系统行为，但不能证明用户价值；PRD 能说明意图，但不一定等于已上线；模型指标也不能代表产品结果。

### 2. 关键洞察

接手的本质不是“读完所有材料”，而是建立一个可以支持决策的产品模型，并明确证据和未知。

### 3. 方案

- 先选择 Product-led、Hybrid audit 或 Evidence recovery 路线。
- 建立证据台账，用 E0–E3 标记置信度。
- 重构产品、用户、旅程、能力和范围。
- 审计模型、Prompt、RAG、Agent、工具、Guardrails 和人工控制。
- 建立业务、产品、AI、安全和系统五层指标。
- 输出产品债、风险、利益相关方议程和 30/60/90 天计划。

### 4. 工程设计

Codex Skill 负责需要上下文判断的工作；Python 标准库负责脚手架、链接、静态站点和验证；离线 UI 负责搜索和交付。这样避免模型重复编写易错构建代码，也不要求目标项目安装依赖。

### 5. 结果与边界

项目有完整测试和 CI，能检查 20 个必需章节、死链、安全形态和离线边界。它不替代真实用户研究、生产数据或利益相关方确认，而是把这些缺口变成明确行动。

## 10 分钟演示顺序

1. **README：** 解释四种断层和核心价值。
2. **SKILL.md：** 展示九阶段接手工作流。
3. **Evidence model：** 展示 E0–E3 和 drift/unknowns。
4. **AI system audit：** 展示从用户意图到反馈评测的链路。
5. **Metrics：** 展示业务→产品→AI→安全→系统的五层分解。
6. **Demo dossier：** 打开 Executive、AI System、Metrics、Risks、30/60/90。
7. **Architecture：** 说明 Skill 判断与 Python 确定性工具的分工。
8. **Tests/CI：** 展示验证器和 GitHub Actions。
9. **Limitations/Roadmap：** 主动说明不能自动获得的真实证据。

## STAR 讲法

- **Situation：** 新 AI 项目交接资料分散，文档、代码和指标口径不一致。
- **Task：** 建立可复用的 AI PM 项目尽调与接手方法，而不是一次性总结。
- **Action：** 设计证据模型、产品/AI/指标框架、固定输出规范、确定性构建器和安全验证。
- **Result：** 形成一个可安装 Skill，自动生成 20 章离线 dossier，并通过测试和 CI。不要虚构业务提升；可以展示覆盖、验证和可复用性结果。

## 常见追问

### 为什么不用 README 或知识库工具？

它们解决内容存储和呈现，不解决证据冲突、置信度、AI 系统审计、指标连接和接手优先级。

### 为什么固定 20 个章节？

为了防止关键决策领域被遗漏。没有证据时保留 E0 页面，比静默缺失更有价值。

### 为什么用 Python，不做完整 Web App？

Skill 需要进入未知项目环境。Python 标准库更容易运行，离线静态站点降低部署和数据泄露风险。Web App 可以作为后续协作版本。

### E0–E3 会不会主观？

会有判断成本，因此定义了证据类型、允许写法和降级条件。目标不是概率精确，而是让不确定性显式、可讨论、可行动。

### 如何评价 AI 产品，而不是模型？

从业务、产品、AI 质量、安全和系统五层建立指标，并要求每个指标有定义、来源、负责人、基线、目标和决策用途。

### 如何处理真实公司保密信息？

内部 dossier 和公开 case study 分开；公开版只保留方法、设计、权衡和学习，移除公司、客户、Prompt、指标、架构和内部路径。

### 这个 Skill 的最大限制？

无法从代码获取真实用户价值、组织决策和生产指标。它会把缺口列为 E0，并生成需要谁、用什么方式确认的议程。

## 面试展示注意事项

- 不要声称工具“自动理解一切”。强调证据和边界。
- 不要虚构业务指标。展示结构覆盖、验证结果和方法复用性。
- 先讲产品问题，再讲 AI 框架，最后讲工程实现。
- 展示一条真实链路：Claim → Evidence → Confidence → Risk/Decision → 30-day action。
- 主动讲 trade-off 和下一步，会比只展示功能更像成熟 AI 产品经理。
